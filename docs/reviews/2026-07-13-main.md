## Summary

This slice wires `/announcements` to the live Nest admin API: a split composer + list page with `?id=` selection, create/update of title/body/region, dedicated publish/unpublish switches, and after-create image upload (upload-url → XHR PUT → attach). The lib layer is strong — every Nest response is runtime-validated with typed guards, MIME/size limits are enforced client-side before requesting an upload URL, and cache writes are consistent. All seven ADR-0012 acceptance criteria are met (live-only, no mock, no client PostHog, create-unpublished, update-never-flips-publish, out-of-scope cuts respected). The headline issue is a form-reset effect in the composer that silently discards unsaved edits whenever the selected announcement's cache entry changes (publish toggle, image upload, or a background refetch) — the exact footgun the Plans dialog deliberately guards against. A secondary error-handling inconsistency and a couple of small a11y/maintainability items round it out.

## Major

### 🟠 Composer reset effect silently discards unsaved edits, `src/components/announcements/AnnouncementsComposer.tsx:65`

**Problem**: The `useEffect` calls `form.reset(...)` on every change of the `selected` prop identity. `selected` is derived in `AnnouncementsView` via `useMemo(() => data.announcements.find(...), [data.announcements, selectedId])`. Any mutation success calls `writeAnnouncementCache`, which `setQueryData`s a brand-new list array (and also `invalidateQueries`), so `data.announcements` — and therefore the `selected` object — gets a new identity even when the id is unchanged. The effect then fires and overwrites the form with server values, wiping in-progress edits regardless of dirty state. Concrete flows that hit this: (a) admin edits the body, then toggles the Publish switch → publish success rewrites the cache → body reverts; (b) admin edits the title, then uploads/replaces an image → attach success rewrites the cache → title reverts. Because `staleTime` is 60s with default `refetchOnWindowFocus`, a plain window refocus mid-edit can also trigger it.

**Why it matters**: Silent loss of a user's typed content during a normal workflow (compose → attach image, or edit → publish). No warning, no dirty guard — the admin re-types work they thought was preserved.

**Suggested fix**: Reset only when the *selected id* actually changes (key the effect by `selected?.id` rather than the whole object), and/or skip the reset when the form is dirty — mirror the Plans dialog, which keys reset by `planId`/`open` and syncs individual fields only when `!dirtyFields[field]` (see `PlanFormDialog.tsx:86-99`).

## Minor

### 🟡 Root error is always set even when field errors were mapped, `src/components/announcements/AnnouncementsComposer.tsx:93`

**Problem**: After a failed submit the code maps Nest field errors and then *unconditionally* calls `form.setError("root", ...)`. Plans and Content only set the root banner when no field errors were mapped (`if (Object.keys(mapped).length === 0)` — see `LessonMediaSection.tsx:196` and `PlanFormDialog.tsx:118`). Here, a `400` with a `title` field error shows both the clean inline field message *and* a redundant root banner carrying the raw internal `ApiError.message` (`[api/apiFetch] POST /announcements/admin returned 400`).

**Why it matters**: Redundant, developer-flavored error text alongside the proper inline message; inconsistent with the established convention for the two sibling features that share this exact helper.

**Suggested fix**: Only set the `root` error when the mapped field-error object is empty, matching the Plans/Content pattern.

### 🟡 Clickable list rows use `aria-selected` on a plain table row, `src/components/announcements/AnnouncementsList.tsx:88`

**Problem**: Each `<tr>` gets `tabIndex={0}`, `onClick`, `onKeyDown` (Enter/Space), and `aria-selected`. `aria-selected` is only valid on rows inside a `grid`/`treegrid` (or on `option`/`tab`/etc.), not a row in a default `table`, so assistive tech won't reliably convey selection, and the row isn't exposed as an interactive control.

**Why it matters**: Selection state and clickability aren't communicated to screen-reader users; keyboard focus lands on an element with no announced role.

**Suggested fix**: Either give the interactive element an appropriate role (e.g. a `role="button"` element / a real button on the subject cell) or model the table as a grid. At minimum drop `aria-selected` from the plain `<tr>` in favor of a supported affordance.

## Nits

- ⚪ `src/components/announcements/AnnouncementsComposer.tsx:95`, `if (mapped)` is always truthy — `nestFieldErrorsFromApiError` returns `{}`, never null; the guard is dead and misleadingly implies a nullable return.
- ⚪ `src/components/announcements/AnnouncementsView.tsx:57` / `AnnouncementsContainer.tsx:36` / `announcements/page.tsx:29`, the same `PageHeader` block is duplicated in three places (Suspense fallback, container Frame, view); consider hoisting into the page/layout so loading, error, and success share one header.
- ⚪ `src/lib/announcements/use-announcement-mutations.ts:170`, upload reuses `putFileWithProgress`, whose thrown `ApiError` messages are prefixed `[content]` — cosmetic, but those strings can surface to an announcements admin.
- ⚪ `src/components/announcements/AnnouncementImageSection.tsx:33`, `useUploadAnnouncementImage` accepts a `signal` but the section never passes one nor aborts on unmount; an in-flight upload can't be cancelled.

## Strengths

- Thorough runtime validation: `parse-announcements.ts` type-guards every Nest DTO (summary, list, upload-url) and throws a typed `ApiError({ kind: "parse" })`, so shape drift fails loudly instead of corrupting the UI.
- Correct, minimal contract adherence: create sends only title/body/region and never touches publish; publish/unpublish go through the dedicated endpoints; image upload validates MIME + 5 MB *before* requesting an upload URL and reuses the shared XHR helper.
- Clean container UX: `errorMessage` maps 404/parse/unauthorized/network/config to human-readable guidance, and loading/empty/error-with-retry states are all present with no mock fallback and no client PostHog — exactly as ADR 0012 requires.

Worth exercising under `/check verify` specifically because typecheck won't catch them: (1) the composer reset behavior — edit a field, then toggle Publish or upload an image, and confirm whether edits survive (the Major above); (2) a `400` submit path to confirm inline field errors render without a redundant raw root banner; (3) keyboard selection of list rows for the a11y note.
